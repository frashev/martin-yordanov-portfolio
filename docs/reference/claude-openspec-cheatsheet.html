◆ OpenSpec commands
openspec init Initialize OpenSpec in the repo — run once after project setup
openspec list List all active changes and their status
openspec validate Validate all artifacts before archiving
/opsx:explore [topic] think Explore an unclear idea — options, risks, no code edits
/opsx:propose <name> plan Create proposal.md, design.md, tasks.md, spec deltas — all in one step
/opsx:new plan Start a change and build its artifacts incrementally (alt. to propose)
/opsx:continue [name] plan Create the next artifact for an in-progress change
/opsx:apply [name] build Implement the approved change task-by-task
/opsx:verify [name] check Confirm implementation matches the change’s artifacts (before archive)
/opsx:sync [name] promote Push a change’s delta specs into the main specs (without archiving)
/opsx:archive [name] done Sync to main specs and close the change

◆ Safety rules
✓
Allow: file reads, git status/diff/log, tests, lint, local dev server
⚠

✗

◆ Full flow — large / architectural change (the maximum; smaller changes use a subset — see “Right-size by volume”)
1
Explore
/opsx:explore
Unclear task? Think before committing. No edits. Get options + risks.
2
Propose
/opsx:propose <name>
Creates proposal.md, design.md, tasks.md, spec deltas. No code yet.
3
Review
(human)
Check scope, non-goals, acceptance scenarios, and task list before approving.
4
Apply
/opsx:apply [name]
Implement task by task. Update tasks.md as you go. Keep diffs small.
5
Run & Verify
/run   /verify
Build, start app, check affected routes, mobile + desktop, browser console.
6
Code Review
/code-review high --fix
Bugs, over-engineering, missing tests, cleanup. Auto-fix what you can.
7
Spec Verify
/opsx:verify [name]
Does the code match what the proposal and spec described? Close the loop.
8
Archive
/opsx:archive [name]
Sync to main specs, close the change. Commit code + artifacts together.
◆ Right-size by volume — match effort to the size of the change
Size How to spot it Recommended flow Enough ✓ Overkill ✗
Tiny 1 file · a few lines · copy, color, spacing, a single value · no new behavior, no new component. edit → npm run build Build / lint green; eyeball it if visual. OpenSpec · /verify · /code-review
Small 1 component or 1 logic file · self-contained tweak · no new route / dependency · fits in one sentence. /plan (if unsure) → edit → npm run dev glance Quick browser look; npm run test if it’s logic. propose→archive cycle · deep /verify · /opsx:verify
Medium 2–8 files · new user-visible behavior · touches shared layout · new tests likely. (motion, SEO/meta, dark mode, gallery lightbox) /opsx:propose → review → /opsx:apply → one runtime check → /codex:review → /opsx:archive ONE runtime proof — /verify or e2e, not both. /run + /verify both · e2e + deep /verify · /code-review ultra
Large Cross-cutting / architectural · new dependency, backend, data model, or auth/security · many files · needs a paragraph to explain. the full 8-step flow above — explore → archive Full triangle (runtime + diff + spec) · + /security-review if auth/data. — this tier earns the ceremony
◆ Estimate the volume in 10 seconds
Files touched 1 → Tiny/Small · 2–8 → Medium · 8+ or new subsystem → Large
User-visible? Nothing new to see/click → Tiny · new behavior → Medium+
New dep / backend / schema / auth? Any “yes” → Large + OpenSpec, always
Explain it in… one sentence → Small · a paragraph → Medium+
Unsure? Size up one tier — under-verifying costs more than the extra step.

✗

◆ Bug fix flow
1. Paste error, log, screenshot, or failing test
2. Reproduce or explain how to reproduce
3. Find root cause — explore before editing
4. Propose smallest safe fix
5. Add or update a regression test
6. Implement
7. Run targeted tests + lint + build
8. /diff Confirm only what was intended changed

Do not refactor unrelated code during a bug fix.

◆ When to use OpenSpec
✓
New feature or changed user-visible behavior
✓
API, data model, auth, or security changes
✓
Cross-cutting refactor affecting multiple files
✓
Anything you will want to understand 6 months later
—
Typos and trivial text changes — skip OpenSpec
—
Small obvious bug fixes with narrow scope
—
Throwaway experiments
◆ Which tool for what
Prompt Quick questions, small single-file edits, explanations
Skill Repeated workflows: review, test gen, PR description, UI polish
Subagent Parallel isolated investigation — agents report back, do not coordinate
Worktree Parallel coding without file conflicts — claude --worktree <branch>
Context7 Fresh library / framework docs — use before coding with any external API
/goal Depth loop — keep retrying until tests pass or metric is met
◆ Verification — three separate checks
/run & /verify runtime Does the app work in a real browser? Routes, nav, mobile, console.
/code-review diff Is the code correct? Bugs, over-engineering, missing tests.
/opsx:verify spec Does the implementation match what was proposed and agreed?

A passing build alone is not enough — routing, mobile behavior, and fallback states are invisible to npm run build.
